二、Agent Loop 核心架构

核心循环极简，不要过度复杂化循环本身：

`python

from zai import ZhipuAiClient

client = ZhipuAiClient(api_key="your-api-key")

messages = [{"role": "user", "content": user_input}]

while True:

response = client.chat.completions.create(

model="glm-5.1",

messages=messages,

tools=tool_definitions, # Function Call 工具定义

tool_choice="auto", # 自动决策是否调用工具

max_tokens=65536,

temperature=1.0, # thinking 模式固定 1.0

)

message = response.choices[0].message

messages.append(message.model_dump())

if message.tool_calls:

for tool_call in message.tool_calls:

result = execute_tool(tool_call.function.name,

json.loads(tool_call.function.arguments))

messages.append({

"role": "tool",

"content": json.dumps(result, ensure_ascii=False),

"tool_call_id": tool_call.id

})

else:

print(message.content)

`

关键原则：新能力只通过三种方式接入——扩展工具集、调整系统提示、状态外化到文件。不要让循环体变成状态机。

三、六大最佳实践

1. 上下文工程：分层管理，防止 Context Rot

上下文越长信号越容易被噪声稀释，按使用频率分层：

层级内容加载策略
常驻层身份定义、项目规则、禁止项每次 session 必加载，保持短而稳定
按需层Skills、领域知识描述符常驻，完整内容触发时注入
运行时层当前时间、用户偏好、动态信息每轮按需拼入
记忆层MEMORY.md 跨 session 事实需要时才读取
系统层确定性逻辑（Linter/Hook）不进上下文

Prompt Cache 要点：GLM-5.1 缓存 TTL 5 分钟，常驻层越稳定缓存命中率越高。动态信息放消息尾部，不破坏前缀匹配。

> 反直觉：稳定的大系统提示比频繁变动的小提示实际成本更低——写入成本只付一次，后续读取折扣可达 90%。

2. 工具设计：ACI 原则

工具设计对成功率的影响比模型本身更大。

三个原则：

① 工具对应 Agent 目标，不对应底层 API

`python

❌ 粒度过细，Agent 要协调多个工具

{"name": "create_file"}, {"name": "write_content"}, {"name": "set_permissions"}

✅ 一个工具搞定一个目标

{"name": "create_script", "description": "创建可执行脚本，自动设置权限",

"parameters": {"path": "...", "content": "...", "executable": true}}

`

② 描述写"何时用"而非"能做什么"，必须加反例

`python

❌ 功能介绍式（路由准确率 53%）

"description": "处理完整的部署流程到生产环境"

✅ 路由条件式 + 反例（准确率 85%）

"description": "Use when deploying to production or rolling back. Don't use for staging or dev environments."

`

③ 错误结构化返回修正建议

`python

❌ Agent 不知道怎么修

return "Error: update failed"

✅ 结构化错误 + 修正建议

return {"error_code": "POST_NOT_FOUND",

"suggestion": "请先调用 list_posts 获取有效的 post_id"}

`

工具数量克制：5 个 MCP 可能带来 ~55K tokens 工具定义，200K 上下文还没对话就用掉近三成。

4. 长任务：状态外化到文件

长任务最常见的失败不是单步报错，而是 session 结束时任务没做完。

推荐模式：

- Initializer Agent（只跑一次）：生成 feature-list.json、初始代码、进度文件

- Coding Agent（循环执行）：从进度文件恢复现场 → 做一个功能 → 跑测试 → 更新状态 → 提交退出

`json

{

"tasks": [

{"id": "1", "desc": "读取配置", "status": "completed"},

{"id": "2", "desc": "修改 schema", "status": "in_progress"},

{"id": "3", "desc": "更新 API", "status": "pending"}

]

}

`

进度放文件，不放上下文。同一时间只有一个 in_progress，完成一步先更新状态再继续。GLM-5.1 的 8 小时长程能力配合这个模式效果最好。

5. 安全边界先于功能

三件事必须先到位：

1. 白名单授权：只有授权用户可触发 Agent

2. 工作空间隔离：shell 操作强制路径检查，越界直接报错

3. 操作审计日志：每次执行记 audit log（append-only JSONL）

4. Prompt Injection 防护：

- 外部内容标注为

- 敏感操作（写数据库、调外部 API）执行前显式确认

- 最小权限——不给 Agent 不需要的工具

6. 评测先行

先修评测，再改 Agent——评测系统出问题和模型退化表现一模一样。

- 有明确正确答案 → 代码评分器（确定性最高）

- 需判断语义质量 → 模型评分器

- 拿不准的 → 人工标注校准

- 从 20-50 个真实失败案例启动，不需要等完整体系

- 通过率接近 100% 时补充更难的任务

六、工程落地推荐顺序

1. 单任务先跑通 —— 一个 Agent Loop + 2-3 个工具，验证核心链路

2. 安全边界先于功能 —— 工作空间隔离、白名单、参数验证

3. 记忆整合早做 —— 不加整合，长对话第 20 轮后基本垮掉

4. Skills 先于新工具 —— 领域知识用文档管理比加新工具更灵活

5. 第一个失败就建评测 —— 不要等积累够了再开始
